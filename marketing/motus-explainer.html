1 What is Motus?

The big picture in 30 seconds

Motus is a physical rehabilitation web app. It connects patients who need to do therapy exercises (like hand rehabilitation after an injury) with the therapists who guide them.

Instead of only doing exercises in a clinic, patients can do them at home, record themselves on video, report their pain level, and have their therapist monitor everything remotely.

Think of it like this: Imagine you broke your finger and need to do specific hand exercises every day to get better. Your therapist gives you a list of exercises through the app, you do them at home while recording a video, and your therapist watches the videos later to make sure you're doing them right -- like a coach watching game film.

Who Built It

Yash Saoji and Oliver Huelsbeck, starting in 2025. It was originally called phalanX (phalanges = finger bones), then rebranded to Motus (Latin for "movement"). The app is fully built and functional but hasn't launched publicly yet.

2 The Two Main Users

Everything in Motus revolves around two people

Patient

The person recovering from an injury

  • Views their assigned exercises
  • Watches demo videos from therapist
  • Records themselves doing exercises
  • Reports pain level (1-10) per set
  • Tracks progress over time
  • Messages their therapist

Therapist

The professional guiding recovery

  • Connects to patients via a clinic code
  • Assigns exercise protocols (what to do, how many reps)
  • Records demo videos showing proper form
  • Reviews patient session videos
  • Writes clinical notes
  • Monitors pain trends and adherence
  • Messages patients

There are also two other roles

Admin -- approves new therapist accounts (created manually in the database, not through the app).

Therapist Pending -- a therapist who signed up but hasn't been approved yet. They're blocked from using the app until an admin says "yes."

9 The Database

What gets saved, where, and why

Think of the database like a filing cabinet. Each drawer (called a "collection") holds a different type of document. When you do something in the app -- log in, record a session, send a message -- it writes a document into the right drawer. When you open a screen, it reads documents from the right drawers to show you your data.

Firestore Collections (The Drawers)

Motus uses Firebase Firestore, a NoSQL database. Data is organized into collections, each containing documents:

CollectionWhat's In ItDocument ID
users Name, role, consent status, language, demographics The user's email
connections List of patient emails linked to this therapist Therapist's email
protocols Exercise assignments (type, reps, sets, frequency, demo video URL) Patient's email
sessions Each exercise session: reps, pain, video URL, date, per-set data Auto-generated
messages Each message: from, to, text, timestamp, read status Auto-generated
clinicalNotes Therapist's notes about a patient (HTML from rich-text editor) Patient's email
therapistLibrary Custom exercises, hidden built-ins, overrides Therapist's email
auditLog Every important action: who did what, when. Append-only (can't delete). Auto-generated
Why is the document ID the user's email? It makes lookups instant. When you log in, the app knows your email, so it can go directly to users/you@email.com without searching through every document. It's like a phonebook organized by name -- you look up "Saoji" and go straight to the right page.

10 How Videos Work

Recording, uploading, storing, and watching

The Video Pipeline

Camera records
Blob created
Upload to Cloudinary
URL saved to Firestore
Therapist watches

The browser's MediaRecorder API captures video from the camera. When recording stops, it produces a "blob" (a chunk of binary data). That blob gets uploaded to Cloudinary via a simple fetch() POST request. Cloudinary returns a URL, which gets saved into the Firestore session document. When the therapist clicks "play," it loads the video from that Cloudinary URL.

Three Types of Videos

TypeQualityMax LengthExpires AfterUsed For
Demo800 kbps (best)2 minutesNeverTherapist showing proper exercise form
Session500 kbps10 minutes14 daysPatient recording their exercise set
Message300 kbps (smallest)1 minute7 daysVideo messages between patient and therapist
Why Cloudinary and not Firebase? Cloudinary is a service specifically designed for video -- it handles compression, format conversion, and fast delivery automatically. Uploading is "unsigned" (no secret key needed in the browser), which keeps things simple. Deleting old videos requires the secret key, so that happens on the server side through a Cloud Function. It's like using YouTube to host videos instead of trying to build your own video player from scratch.

Cloud Function: Video Expiry

A scheduled Cloud Function called expireVideos runs once per day. It:

  1. Finds all session documents where videoExpireAt is in the past
  2. Calls the Cloudinary API (with the secret key) to delete the video file
  3. Sets videoUrl to null in Firestore so the app knows it's gone

12 Messaging System

Simple, real-time text messaging

Motus has built-in messaging between patients and their connected therapist. It works like a simple text chat.

Patient types message + hits SendsendMessage(from, to, text)
Document written to Firestoremessages collection: from, to, text, timestamp, read status
Therapist's app detects new messageFirestore real-time listener auto-updates the thread

Messages are organized into threads. A thread ID is just both email addresses sorted and joined with a colon (e.g., doctor@clinic.com:patient@gmail.com). Each message tracks who has read it. Unread counts show as badges.

How real-time works: Firestore has a feature called "listeners" -- the app subscribes to changes in the messages collection, and whenever a new document appears, the screen updates automatically. No refreshing needed. It's like having a mailbox that rings a bell every time a new letter arrives.

13 Keeping Things Safe

HIPAA compliance, privacy rules, and data protection

Why This Matters More Than Most Apps

Motus handles Protected Health Information (PHI) -- things like medical videos, pain levels, injury details, and therapist notes. In the United States, health data is protected by a law called HIPAA. Breaking HIPAA rules can result in massive fines. So Motus has to be much more careful than a typical app.

Security Layers

LayerWhat It DoesHow
Firestore RulesControls who can read/write what dataPatients can only see their own data. Therapists can only see data for patients connected to them. Audit log is append-only.
App CheckBlocks fake/bot requestsreCAPTCHA v3 in production verifies the request comes from a real browser, not a script.
XSS ProtectionPrevents code injectionAll user-generated text is escaped before rendering. No raw innerHTML with user content.
Input ValidationRejects bad dataLength limits, type checking on all form inputs.
Session TimeoutAuto-logout on idle20 minutes of no clicks = logged out. Warning at 18 min.
Audit LoggingRecords every important actionWho did what, when. Can never be deleted or modified.
PHI ScrubbingKeeps health data out of error reportsSentry replaces email addresses with [email] before sending.
ConsentLegal acknowledgmentPatients must read and accept the HIPAA NPP before accessing any screen.
Data ExportHIPAA Right of AccessCloud Function returns all of a patient's data as JSON on request.
Account DeletionHIPAA Right to DeleteCloud Function deletes everything: user doc, sessions, messages, videos, Firebase Auth account.
Automated BackupsDisaster recoveryDaily Firestore export to Google Cloud Storage. 90-day retention.
Video ExpiryMinimizes stored PHISession videos auto-delete after 14 days. Message videos after 7 days.
Think of Firestore rules like a bouncer at a club. Every time someone tries to read or write data, the bouncer checks: "Are you logged in? Is this YOUR data? Are you a therapist connected to THIS patient?" If any answer is no, the request is blocked -- the code never even runs. This happens at the database level, so even if someone tried to hack the JavaScript, the database itself would reject the request.

Sections 14 and 16 don't exist (skipped in numbering). Sections 9, 11, 13, and 17 are dormant -- their code is fully written but only activates when ANGLE_TRACKING_ENABLED = true.

Why one giant file instead of many small ones? This was a deliberate decision (documented as Decision D-1). For a two-person team using AI-assisted coding, having everything in one file makes it easy to find things and make changes that touch multiple parts of the app. It's like keeping all your homework in one binder instead of 17 different folders -- less organized, but nothing ever gets lost.

The Window Exports Trick

Because app.js is loaded as an ES module (with type="module"), functions inside it are private by default. HTML buttons use onclick="functionName()", which only works if the function is a global variable on window.

Solution: at the very end of app.js, there's a big Object.assign(window, { ... }) block that explicitly exposes every function that HTML buttons need to call. If you write a new function and forget to add it here, clicking the button will do nothing.
