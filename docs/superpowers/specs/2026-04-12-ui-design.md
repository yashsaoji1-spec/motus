# Motus UI Design Spec
**Date:** 2026-04-12  
**Authors:** Oliver Huelsbeck, Yash Saoji  
**Status:** Approved — ready for implementation

---

## 1. Design Philosophy

Motus has two distinct user roles with fundamentally different needs. The UI reflects this with two intentional visual systems that share a codebase but feel like purpose-built products:

- **Patient side** — warm, motivating, fitness-app energy. Patients use this daily during recovery. It must feel encouraging, celebratory, and effortless.
- **Therapist side** — clinical, efficient, EHR-like. Therapists are power users scanning multiple patients. It must be dense, fast to read, and distraction-free.

**Core constraint:** Minimize scrolling on all screens, especially patient screens. Every patient screen must fit in a single viewport on a standard mobile device.

| Token | Value | Usage |
|---|---|---|
| `--danger` | `#CC2936` | Pain indicators, errors, destructive actions |
| `--success` | `#10B981` | Confirmed states, completed badges |
| `--gold` | `#F59E0B` | Streaks, achievement highlights |
| `--font-mono` | `'DM Mono', monospace` | Rep counts, pain scores, timestamps, data values |

### 2.4 Typography Scale

**Patient (Nunito):**
- Hero heading: `2rem / 800` — greeting on home screen
- Title: `1.2rem / 700` — screen titles
- Body: `0.9rem / 400` — card content
- Label: `0.72rem / 700 / uppercase / 0.07em tracking` — section labels, tags

**Therapist (IBM Plex Sans):**
- Title: `1.1rem / 600` — patient name in detail panel
- Body: `0.85rem / 400` — list content
- Meta: `0.72rem / 500` — timestamps, compliance stats
- Label: `0.65rem / 600 / uppercase / 0.08em tracking` — section labels

**Monospace (DM Mono):** Used across both sides for all numeric data — rep counts, pain scores, session counts, times.

## 4. Layout System

### 4.1 Patient Layout — No-Scroll Rules

Every patient screen uses a **fixed 3-zone column layout**:

```
┌─────────────────────┐
│   Header Zone       │  fixed height — gradient bg, greeting, streak
├─────────────────────┤
│   Hero CTA Zone     │  fixed height — primary action always visible
├─────────────────────┤
│   Content Zone      │  flex: 1 — fills remaining viewport
└─────────────────────┘
```

Rules:
1. **Hero CTA is always visible** on the home screen. It never scrolls out of view.
2. **Sub-screens open as full-screen overlays** — Progress, Messages, Exercises replace the current view entirely. Nothing stacks below the fold.
3. **Session flow is step-based** — one step per screen (exercise select → recording → set input → complete). No long forms.
4. **Set input uses a bottom sheet** — slides up over the camera view. No navigation change.
5. **2×2 nav grid uses `flex: 1`** — expands to fill remaining height on any phone size. No fixed heights that cause overflow.
6. **Pain scale is a tap bar** — 10 colored segments (green → amber → red), no number keyboard.

### 4.2 Therapist Layout — Three-Column

```
┌──────┬───────────────┬──────────────────────────┐
│ Icon │  Patient List │  Patient Detail Panel    │
│ Nav  │  (scrollable) │  (fixed height, tabbed)  │
└──────┴───────────────┴──────────────────────────┘
```

Rules:
1. **Icon sidebar is fixed** — never scrolls, always visible. Active section highlighted with `--th-primary`.
2. **Patient list is the only scrollable column** — the detail panel and sidebar do not scroll.
3. **Patient detail uses tabs** — Overview / Sessions / Protocol. No vertical scroll needed for normal patient loads.
4. **Actions open as modals** — Add Protocol, Bulk Assign, Send Message open as centered modals, not inline forms that push content down.
5. **Patient selection slides in detail panel** — no page navigation, instant swap.

### 4.3 Header Treatment

Both sides share a consistent header pattern:

- **Background:** Deep navy (`#0C4A6E`) blending to sky blue via gradient
- **Wave SVG separator:** Transitions header into the body background — no hard edge
- **Back navigation:** Text link top-left (`← Back`), never a button

## 6. Screen Specifications

### 6.1 Auth Screens (loginScreen, signupScreen, forgotScreen)

**Shared header treatment:**
- Gradient background: `linear-gradient(160deg, #0C4A6E 0%, #0EA5E9 100%)`
- Login: includes logo mark (placeholder icon + "Motus" wordmark + tagline)
- Signup/forgot: header is shorter, shows screen title only

**Login form:**
- Email input → Password input → Sign In (gradient CTA) → Create Account (outline) → Forgot password (text link)
- No role selector on login — role is determined from Firestore on auth

**Signup form:**
- Role segmented control (Patient / Therapist) — pill shape, active tab filled with `--pt-primary`
- Full name → Email → Password → Create Account (gradient CTA)
- "Already have an account?" text link at bottom

**Both screens:** No scroll on standard phone size. All elements fit in viewport.

### 6.2 Consent Screen (consentScreen)

- Same gradient header, shorter
- Single white card containing privacy + video recording consent text
- Card fills available height — text is not truncated, no internal scroll on standard phones
- Single full-width "I Agree — Continue" gradient CTA at bottom

### 6.3 Patient Home (patientScreen)

**Zone 1 — Header (fixed):**
- Deep navy background (`#0C4A6E`)
- Left: "Good morning" label (sky blue, small caps) + patient first name (white, 800 weight, large)
- Right: Streak badge (gold background, flame icon placeholder, streak count)
- Wave SVG separator at bottom of header

**Zone 2 — Hero CTA (fixed):**
- Full-width gradient card (`--pt-cta-gradient`)
- Left: exercise name + set count ("Finger Extension · 3 sets")
- Right: play button circle (semi-transparent white)
- If no session due today: card shows "No session today" in muted style

**Zone 3 — 2×2 Nav Grid (flex: 1):**
- My Progress / Messages / My Exercises / Profile
- Each cell: white card, rounded, icon placeholder, label
- Messages cell: green dot badge overlay on icon when unread messages exist
- Grid uses `display: grid; grid-template-columns: 1fr 1fr; gap: 8px`

### 6.4 Exercise Select Screen (exercisesScreen)

- Same header treatment as home, back link top-left
- List of assigned protocols — one card per protocol
- Each card: exercise name, set/rep/frequency summary, completion badge (X/Y done)
- Completion badge: green (`--pt-accent`) if in progress, sky blue if not started
- "Start Selected" gradient CTA pinned to bottom — always visible, no scroll needed for ≤4 protocols

### 6.5 Session Recording Screen (manualCamScreen)

**Camera area (flex: 1, dark):**
- Background: `#0F172A` — near black for viewfinder contrast
- Top HUD: back link (left) + REC timer badge (right, emerald border + `DM Mono` time)
- Bottom HUD: segmented progress bar (one segment per set, filled = completed), set label

**Control card (fixed bottom):**
- Background: `#0C4A6E` (navy)
- Exercise name + set label (left) + elapsed time in `DM Mono` (right)
- Full-width "End Set" button (emerald solid)

**Recording state:**
- Segment bar fills as sets complete
- Timer counts up in `DM Mono`
- No other controls visible — no distraction during recording

### 6.6 Set Input Bottom Sheet

Slides up over the camera view. Components in order:
1. Drag handle
2. "Set N complete" heading (Nunito 800)
3. Reps completed — large `DM Mono` input, pre-filled with protocol rep count
4. Pain level label
5. 10-segment color bar: segments 1–3 emerald, 4–6 amber, 7–10 danger — tapped segment highlighted with dark border
6. Optional notes input (single line, collapsed by default)
7. "Save + Continue" gradient CTA (or "Save + Finish" on last set)

### 6.7 Session Complete Screen

Centered layout, no header zone:
- Large gradient circle with checkmark (64px)
- "Session done!" heading + summary line ("3 sets · avg pain 4/10")
- 3-stat row: sets done / streak / avg pain — each in a white card with `DM Mono` value
- "Back to Home" gradient CTA — full width
- Nothing to scroll

### 6.8 Progress Screen (progressScreen)

- Same header treatment, "My Progress" title
- Pain trend chart (Chart.js) — fixed height, sky blue bars fading to emerald as pain decreases
- Session history below chart — day-grouped cards
- This is the one patient screen where scrolling is acceptable (session history can be long)
- Chart is always visible above the fold; history scrolls beneath it

### 6.9 Messaging Screen (messagingScreen)

**Patient view:**
- Same navy header, therapist name as subtitle
- Message bubbles: incoming = white card with sky border; outgoing = `--pt-cta-gradient`
- Read receipt ("Read") shown below outgoing messages in muted small type
- Input bar pinned to bottom: text input + gradient send button
- Message thread scrolls; input is always visible

**Therapist view:**
- Opens as a panel in the detail area (no separate screen)
- Same bubble style but using therapist tokens (blue outgoing, white incoming)

### 6.10 Therapist Dashboard (therapistScreen)

**Sidebar (48px, fixed):**
- Background: `--th-sidebar-bg` (`#1E293B`)
- Logo mark at top (gradient circle)
- 4–5 icon slots (filled placeholder boxes) — Patients, Messages, Library, Clinic, Settings
- Active icon: `--th-primary` background
- Inactive icons: `rgba(255,255,255,0.12)` background

**Patient List (110px, scrollable):**
- Background: white
- Search input at top (slate background, rounded)
- Patient rows: `--th-accent-border` for active patients, `--th-muted` border for new/inactive
- Selected patient row: `#EFF6FF` background

**Detail Panel (flex: 1):**
- Header: patient name (strong) + compliance summary (muted)
- Tabs: Overview / Sessions / Protocol
- Overview tab: pain trend bar chart (fixed height) + recent session rows
- Sessions tab: full session history with per-set data (this tab scrolls)
- Protocol tab: current protocol items + Add Protocol button
- "Add Protocol" button always visible at bottom of panel

### 6.11 Admin Screen (adminScreen)

- Uses therapist tokens (clinical feel appropriate)
- Simple list of `therapist_pending` accounts
- Approve / Reject actions per row — inline, no navigation
- No scroll needed for small pending lists

### 6.12 Connect Screen (connectScreen)

- Patient-scoped (uses patient tokens)
- Single input for clinic code or therapist email
- "Connect" gradient CTA + "Skip for now" text link below
- No scroll

## 7. Navigation Map

### Patient Navigation
```
loginScreen / signupScreen
    └── consentScreen (first login only)
        └── connectScreen (no therapist yet)
            └── patientScreen (home)
                ├── exercisesScreen (→ manualCamScreen → sessionComplete)
                ├── progressScreen
                ├── messagingScreen
                └── Profile (bottom sheet — not a separate screen)
```

Patient sub-screens (Progress, Messages, Exercises) use full-screen overlay transitions. Profile opens as a bottom sheet over the home screen. Back navigation always returns to `patientScreen`.

### Therapist Navigation
```
loginScreen
    └── pendingScreen (if therapist_pending)
    └── therapistScreen (single screen, panel-based)
        ├── Patient detail panel (inline)
        ├── Messaging panel (inline)
        ├── Library modal
        └── Clinic screen (full-screen overlay)
```

## 8. Implementation Approach

**Approach A — Design system first, screens second.**

1. Purge existing CSS variables. Replace with the full token system from Section 2.
2. Add `.patient-scope` and `.therapist-scope` wrappers to `index.html`.
3. Update Google Fonts import.
4. Implement shared globals (danger, success, gold, DM Mono).
5. Build patient component classes (cards, buttons, inputs, tags, bottom sheet).
6. Build therapist component classes (cards, buttons, inputs, labels, tabs).
7. Implement screens in this order:
   - Auth screens (login, signup, forgot) — shared baseline
   - Patient home
   - Session flow (exercise select → recording → set input → complete)
   - Patient sub-screens (progress, messaging)
   - Therapist dashboard
   - Remaining screens (connect, consent, admin, clinic)

No screen is touched until the token layer is complete. Screens are implemented using tokens only — no hardcoded color values in screen CSS.

## 9. What Is Not Changing

- Single-file architecture (`index.html`, `app.js`, `styles.css`) — no splitting
- All JS logic — this spec covers HTML/CSS only
- Firebase collections, auth flow, or data model
- MediaPipe on CDN
- `ANGLE_TRACKING_ENABLED` feature flag behavior
- Window exports block in `app.js`
