# Motus UI Clarity Findings Register

Severity: **S1** blocks a journey / data loss · **S2** likely to confuse a non-technical user · **S3** polish/inconsistency.
Source: `gates` (axe) · `journey` · `i18n` · `reading` · `heuristic` (manual review).

---

## Summary

**35 findings total: 3 S1, 24 S2, 8 S3.**

**S1 (journey blockers, 3):** All three are genuine failures from the journey suite — J2 (no visible save affordance on manualCamScreen), J3 (icon-only send button with no accessible name), J5 (no success feedback after admin approval). J3 is corroborated across two sources: `journey` (timeout waiting for `button[name=/send/i]`) and `gates` (`messagingScreen @ */: button-name` critical axe violation across all viewports).

**S2 (likely to confuse non-technical users, 24):** Dominant cause is a single design token (`#0ea5e9` sky-500 on `#f0f9ff` sky-50, contrast ratio 2.59:1 vs 4.5:1 required) hitting 8 screens across all 3 viewports. Additional S2 findings: `list` structure on patientScreen, 10 missing async loading/error/success states across 5 action functions, high reading-level on clinically important copy (consent, HIPAA, therapist-wait notice), and no user-facing error if `saveProtocol`, `loadClinicScreen`, or `approveTherapist` throw.

**S3 (polish/consistency, 8):** 50 hardcoded HTML strings not routed through the i18n key system (en/es parity is complete, so this is technical debt rather than an active gap), and the bulk of the 62 above-grade-6 strings are single-word labels or placeholders that are FK scoring artifacts.

Most-affected screens by finding count: **messagingScreen** (3 findings: J3/gates button-name S1, sendMessage no loading S2, sendMessage no error S2), **adminScreen** (3: J5 S1, contrast S2, approveTherapist no loading/success S2), **manualCamScreen** (3: J2 S1, finishManualSession no loading S2, finishManualSession no success S2).
